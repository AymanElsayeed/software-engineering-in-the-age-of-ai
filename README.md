# Software Engineering in the Age of AI

This repository contains tasks, experiments, and implementation artifacts from the MSE course
**"Software Engineering in the Age of AI."**

---

## Purpose

This repository documents a research-based exploration of integrating non-deterministic AI into deterministic software engineering systems.

The focus is not just "using AI tools better," but treating AI as an **architectural component** that must be designed, constrained, monitored, and governed through rigorous engineering patterns.

Core objective:
- Build systems that preserve **stability, reliability, and cost control** while leveraging AI capabilities.

---

## The Core Framework: The 4 Cs

### Collaborate
- Treat AI as an **agent** in a human-led multi-agent system.
- Keep the engineer as the **Navigator** and final decision-maker.

### Communicate
- Move from basic prompting to **engineering prompting**.
- Apply structured techniques such as **Few-Shot**, **Chain-of-Thought**, and deterministic **JSON API** output contracts.

### Construct
- Use resilient architecture patterns such as the **Adapter Pattern** to reduce vendor lock-in and isolate model/API failures.
- Use **RAG (Retrieval-Augmented Generation)** to ground AI behavior in trusted organizational data.

### Control
- Apply **Shift-Left Security** practices early in development.
- Actively manage rising **technical debt** in AI-assisted codebases.
- Use delivery and reliability metrics (including **DORA**) to validate quality outcomes.

---

## Topics Covered

- **Architectural Patterns:** Adapters for failover and provider abstraction, plus semantic caching for lower cost and latency.
- **Grounding AI:** RAG as a solution to the "knowledge gap," separating reasoning logic from knowledge storage.
- **Quality and Maintenance:** Tracking **Change Failure Rate** and **MTTR (Mean Time to Recovery)** to protect stability.
- **Human-in-the-Loop Models:** Distinguishing being *in the loop*, *on the loop*, and *in command*.
- **Engineering Prompting:** Designing prompts as repeatable, testable interfaces rather than ad hoc chat interactions.

---

## Security, Compliance, and Risk

Key technical risks addressed in this repository include:

1. **Prompt Injection**
   - **Direct injection:** Malicious user-provided instructions.
   - **Indirect injection:** Hidden instructions embedded in external data sources (for example, emails or documents).

2. **Secret Management**
   - Never hardcode secrets (API keys, tokens, credentials) in source code.
   - Use `.env` files and automated scanners such as **GitLeaks**.

3. **Compliance by Design**
   - Engineer systems with regulatory controls in mind, including **GDPR**, **SOC 2**, and the **EU AI Act** risk model.

---

## Professional Responsibility

The software engineer remains **100% legally and ethically responsible** for all shipped code, including code generated by AI.

Given the known vulnerability rate in AI-generated outputs, the engineer's role is not only to produce code quickly, but to **validate, secure, and continuously monitor** it before release.
